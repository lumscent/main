# Interface Rules

Статус: `draft`  
Раздел: `design-system`

---

## Назначение

Файл описывает базовые правила проектирования интерфейсов для продуктовых дизайнеров.

Используется при:

- создании новых экранов;
- пересборке существующих интерфейсов;
- дизайн-ревью;
- постановке задач дизайнеру;
- подготовке макетов к разработке;
- проверке UI-решений через Claude или другой AI-инструмент.

---

## Базовые правила интерфейса

| Правило | Что означает |
|---|---|
| У экрана должна быть понятная цель | Пользователь должен быстро понять, зачем он находится на экране |
| Главное действие должно быть заметным | Основной CTA не должен теряться среди вторичных действий |
| Вторичные действия не должны конкурировать с основным | Дополнительные кнопки и ссылки должны иметь меньший визуальный вес |
| Пользователь должен понимать следующий шаг | После действия должно быть понятно, что произойдёт дальше |
| Ограничения нужно объяснять | Если действие недоступно, нужно показать причину |
| Состояния нужно проектировать заранее | У экрана должны быть default, loading, empty, error, success и disabled states |
| Ошибки должны помогать восстановиться | Error state должен объяснять проблему и давать действие |
| Empty state должен вести дальше | Пустой экран не должен выглядеть как тупик или поломка |
| Интерфейс должен учитывать роли | Разные роли могут видеть разные данные, действия и ограничения |
| Не нужно прятать критичную информацию в tooltip | Tooltip подходит для пояснений, но не для информации, без которой нельзя принять решение |

## Проверка экрана перед дизайн-ревью

Перед ревью дизайнер должен проверить:

- понятна ли цель экрана;
- виден ли основной CTA;
- не конкурируют ли действия между собой;
- понятно ли, что произойдёт после клика;
- есть ли все обязательные состояния;
- понятны ли ошибки;
- есть ли пустое состояние;
- есть ли loading state;
- есть ли disabled state;
- учтены ли права и роли пользователя;
- нет ли лишней информации, которая мешает основному сценарию;
- можно ли передать макет в разработку без устных объяснений.

---

## Правила для действий

Основное действие должно:

- быть визуально заметным;
- находиться в ожидаемом месте;
- иметь понятный текст;
- вести к ожидаемому результату;
- не менять смысл в разных частях продукта.

Вторичные действия должны:

- быть менее акцентными;
- не отвлекать от основного сценария;
- быть сгруппированы логично;
- не создавать ощущение одинаковой важности со всеми действиями сразу.

Опасные действия должны:

- быть визуально отделены;
- требовать подтверждения, если действие необратимо;
- объяснять последствия;
- не располагаться случайно рядом с частыми действиями.

---

## Правила для контента

Текст в интерфейсе должен:

- быть коротким;
- объяснять действие или состояние;
- не использовать внутренний жаргон без необходимости;
- помогать пользователю принять решение;
- не обвинять пользователя в системных ошибках;
- быть одинаковым по смыслу в похожих сценариях.

```text
Ошибка. Недоступно.

## Правила для форм

Формы должны:

запрашивать только необходимые данные;
явно показывать обязательные поля;
объяснять, зачем нужны чувствительные данные;
сохранять введённую информацию при ошибке;
показывать ошибки рядом с проблемным полем;
быть удобными на мобильных устройствах;
не требовать регистрации без объяснения ценности.
Правила для списков и карточек

## Списки и карточки должны:

помогать быстро сравнивать элементы;
показывать ключевую информацию до перехода в детали;
не перегружать пользователя второстепенными данными;
иметь понятные действия;
сохранять одинаковую структуру для похожих объектов.

## Для вакансий важно показывать:

название вакансии;
работодателя;
город или локацию;
зарплату, если доступна;
график;
формат работы;
основной CTA.
Правила для состояний

Каждое состояние должно отвечать на три вопроса:

Что произошло?
Почему это произошло?
Что пользователь может сделать дальше?

## Обязательные состояния:

default;
loading;
empty;
error;
success;
disabled;
no results;
permission denied.
